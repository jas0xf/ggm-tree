02Inside the two primitives 03 / 07

Same job, opposite design.

PRG-1 · hardware-accelerated

AES-128

opSubBytes
round01

A standardized block cipher. State is a 4×4 byte grid; each round applies SubBytes → ShiftRows → MixColumns → AddRoundKey. Intel and AMD expose this as a native instruction (AES-NI).

block128 bits
rounds10 byte-level
siliconAES-NI ✓
PRG-2 · lightweight permutation

Spongent π[176]

opAddCounter
round01

A 176-bit permutation designed for hardware area, not throughput. Each round: lFSR counter XOR → PRESENT 4-bit S-box → a bit-permutation j → (j·44) mod 175. No CPU silicon.

state176 bits
rounds80 bit-level
siliconnone
Same 128-bit output, but Spongent does 8× more rounds, each on individual bits.
FIPS-197 · Bogdanov et al., CHES 2011 scroll →

05Performance vs security 06 / 07

Faster kernels leak more timing.

AES kernel choice

S-box · T-tables · bitsliced

T-tables push 4 KB of round tables into shared memory and replace nine S-box loads with four XORs — fast, but introduces cache-bank timing dependent on plaintext. Bitslicing is constant-time by construction; we time-boxed it.

S-box30M
T-tables50M
fixed-key30M
all variants bit-exact · leaves/s @ d=20
lightweight ≠ fast

Spongent's area-first design loses on commodity GPUs

Spongent's 80 bit-level rounds are great for chip area in sensors and RFID tags — but byte-level AES with hardware support still wins by 25× on commodity silicon. "Lightweight" is a hardware-engineer's word.

when CPU wins

Below the crossover, launch overhead dominates

For d ≲ 10 (under 1k leaves) CPU AES-NI matches or beats the GPU because kernel launch ≈ 20 μs swamps the work. OpenMP collapses at t = 16 on this 8-core box — a clean oversubscription lesson.
