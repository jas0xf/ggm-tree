04PRG-1 · inside AES-128 05 / 16

AES-128: ten rounds on a 4×4 byte grid.

operation · SubBytes round 01 / 10

AES treats its 128-bit state as a 4×4 grid of bytes. Each round mutates the grid with four primitives, ending with a key XOR. The 10th round skips MixColumns. Standardized by NIST in 2001 (FIPS-197); shipped in silicon by Intel since Westmere (2010).

ii.ShiftRows — rotate rows by 0, 1, 2, 3 bytesdiffusion
iii.MixColumns — mix each column in GF(28)diffusion
iv.AddRoundKey — XOR with the 128-bit round keykey mixing

05How we implemented AES on the GPU 06 / 16

Three kernel variants, three different trade-offs.

v1 · simple

S-box in __constant__

256-byte S-box · broadcast cache

256-byte lookup

Round keys live in per-thread registers; the S-box is uploaded once into __constant__ memory. Each round costs 9 SubBytes loads + 4 MixColumns multiplications. Simple, correct, slow.

throughput @ d=20 30M/s
timing: constant if S-box index uniform
v2 · fast

T-tables in __shared__

4 × 256 × 4 B · 4 KB per block

4 tables × 256 × 4 B

Precomputed T0..T3 tables collapse SubBytes + ShiftRows + MixColumns into four 32-bit XORs per round. ~1.7× faster than S-box. Loaded into shared memory per block at kernel entry.

throughput @ d=20 50M/s
timing: depends on plaintext bytes
v3 · constant-time

bitsliced · 32-wide

warp-level SIMD · 32 blocks/warp

32 lanes across a warp

Käsper–Schwabe bitslicing: pack 32 AES states across a warp; each round becomes a few hundred Boolean ops. Constant-time by construction. Implementation is heavy; time-boxed (currently falls back to T-tables).

target throughput ~80M/s
timing: cipher-independent

Engineering rule of thumb: for GPU AES, prefer T-tables for throughput and bitsliced for any setting where the input is attacker-controlled.

Käsper & Schwabe · CHES 2009 · "Faster & Timing-Attack Resistant AES-GCM" scroll →

09CPU baselines 10 / 16

Three ground-truth CPU backends.

v1

S-box reference

cpu_1t
1 thread

Direct FIPS-197 implementation: byte-by-byte ShiftRows, xtime-based MixColumns, no SIMD. Single thread. This is the bit-exact oracle every other backend matches against.

@ d=20 · leaves/s 1.5M
v2 · hardware

AES-NI intrinsics

cpu_aesni
1 thread + silicon

_mm_aeskeygenassist_si128 for the schedule, _mm_aesenc_si128 for the inner rounds, _mm_aesenclast_si128 for the last. Single instruction = one whole round. Still single-thread — and still beats CPU OpenMP at high depth.

@ d=20 · leaves/s 8M
v3 · multicore

OpenMP wrapper

cpu_omp · t=1,2,4,8,16
8 threads

One #pragma omp parallel for at each level loop; static schedule; uses the same S-box reference per thread. Scales well to 8 threads on this 8-core box, then collapses at t=16 from oversubscription.

@ d=20 · t=8 · leaves/s 5M

  block = min(256, n)

Each thread reads one parent, computes G, writes two children. At d = 20 that is ~1 M independent threads per launch.

11GPU memory layout 12 / 16

Every byte lives where it should.

tier 1 · fastest

registers

per-thread · ≈100 cycles latency saved

AES round keys (176 B/thread, variable-key); Spongent state (3 × uint64). Recomputed for each parent → cheap and contention-free.

~250 B/thread
tier 2 · per-block

__shared__

256-thread block · same-clock access

AES T-tables — 4 × 256 × 4 B = 4 KB, loaded once at kernel entry. Avoids the broadcast-serialization issue of __constant__ on plaintext-dependent indices.

4 KB
tier 3 · uniform reads

__constant__

all threads broadcast read

S-boxes (AES 256 B, Spongent 16 B), AES Rcon (11 B), pLayer table (176 B), fixed-key round-key schedule (176 B). Always tiny.

≈ 800 B
tier 4 · the tree

global

all threads · BFS array

Flat array of nodes: parent at i, children at 2i+1, 2i+2. Coalesced 32-thread reads (32 × 16 B = 512 B). At d=24, 512 MB.

up to 512 MB

Every __constant__ upload is done once at module-load time and reused for the whole sweep. The global tree buffer is allocated per depth with cuMemAlloc.

14Performance vs security 15 / 16

Faster kernels leak more timing.

AES kernel choice

S-box · T-tables · bitsliced

T-tables push 4 KB of round tables into shared memory and replace nine S-box loads with four XORs — fast, but introduces cache-bank timing dependent on plaintext. Bitslicing is constant-time by construction.

S-box30M
T-tables50M
fixed-key30M
all variants bit-exact · leaves/s @ d=20
lightweight ≠ fast

Spongent's area-first design loses on commodity GPUs

Spongent's 80 bit-level rounds are great for chip area in sensors and RFID tags — but byte-level AES with hardware support wins by 25× on commodity silicon. "Lightweight" is a hardware-engineer's word.

AES GPU50M

when CPU wins

Below the crossover, launch overhead dominates

For d ≲ 10 (under 1k leaves) CPU AES-NI matches or beats the GPU because kernel launch ≈ 20 μs swamps the work. OpenMP collapses at t = 16 on this 8-core box — a clean oversubscription lesson.

15Take-away & future work 16 / 16

Three findings, three open ends.

i.

The tree maps cleanly onto per-level GPU kernels.

found · GPU T-tables = 50M leaves/s @ d=20

Open: subtree-per-block kernels could reduce launch overhead at small d.

ii.

Lightweight crypto is not always fast on commodity GPUs.

found · AES 50M vs Spongent 2M leaves/s

Open: bit-sliced Spongent across a warp could close the gap.

iii.

Faster kernels leak more timing — choose carefully.

found · T-tables ⊃ bank-pattern side channel

Open: ship warp-level bitsliced AES for constant-time GGM.
