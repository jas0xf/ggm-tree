Background

What is a GGM tree?

Goldreich, Goldwasser, Micali (1986): build a PRF from any length-doubling PRG.

G : {0,1}^n → {0,1}^{2n},  split as G(s) = (G_0(s), G_1(s))

For input x = x_1 x_2 … x_d :
  F_K(x) = G_{x_d}( G_{x_{d-1}}( … G_{x_2}( G_{x_1}(K) ) … ) )

Equivalently: walk the binary tree of depth $d$ from root $K$, choosing the $x_i$-th child at level $i$.

Full expansion: $2^{d+1} - 1$ nodes × 16 B. At $d{=}24$ that's 512 MB.

PRG choice 1

AES-128 as a PRG

Variable-key:   G(s) = AES_s(0) || AES_s(1)
                (s is reinterpreted as a fresh AES key per node)

Fixed-key:      G(s) = AES_K(s) || AES_K(s ⊕ 0x01)
                (public K, round keys precomputed once)
  • Variable-key: canonical GGM-from-PRF reading; one key schedule per node
  • Fixed-key: faster on GPU; security argument is PRP-based
  • CPU has hardware support: AES-NI instructions (_mm_aesenc_si128, _mm_aesenclast_si128)

PRG choice 2

Spongent-π[176] as a PRG

Bogdanov et al., CHES 2011 — lightweight hash designed for hardware area.

G(s) = π[176](s || 0x00)[0..15]  ||  π[176](s || 0x01)[0..15]

π[176]: 176-bit state, 80 rounds, each round:
  1. AddCounter   (7-bit LFSR XOR at fixed positions)
  2. SBoxLayer    (PRESENT 4-bit S-box per nibble)
  3. pLayer       (j → (j × 44) mod 175)

No CPU hardware support; everything bit-level.

BFS layout: parent at index $i$, children at $2i+1, 2i+2$ — naturally coalesced (16 B × 32 threads = 512 B contiguous reads/writes per warp).

Implementation

Per-level kernel launch

For ℓ = 0 … d − 1:
  total_threads = 2^ℓ
  block         = min(256, 2^ℓ)
  kernel<<<ceil(2^ℓ / block), block>>>(tree, ℓ)

Each thread:
  parent = tree[(2^ℓ - 1) + i]
  (c0, c1) = G(parent)
  tree[(2^(ℓ+1) - 1) + 2i]     = c0
  tree[(2^(ℓ+1) - 1) + 2i + 1] = c1

Trivially correct, exposes the "GPU underutilized at small ℓ" story.

Verification

Bit-exact cross-backend

  • FIPS-197 Appendix C.1 vector → all AES backends
  • Frozen 31-node GGM-AES tree at $d{=}4$, seed = FIPS-197 key
  • Frozen 31-node GGM-Spongent tree at $d{=}4$
  • Every backend × kernel × thread-count matches byte-for-byte

All 35 pytest cells pass on the RTX 3060: 13 CPU + 4 Spongent + 18 GPU.

Spongent constants are paper-cross-validation TODO (we have internal consistency only).

Results

AES throughput vs depth

  • GPU T-tables overtakes CPU AES-NI around $d{\approx}10$
  • CPU AES-NI cache cliff past $d{=}14$ (tree exceeds L3)
  • CPU OpenMP $t{=}16$ on 8-core box: oversubscribed, throughput collapses

Results

AES vs Spongent on GPU

Gap is consistent at ~25× across all depths.

Spongent's design optimizes for hardware area, not commodity-software throughput. 80 bit-level rounds >> 10 byte-level AES rounds.

Discussion

Trade-offs and side-channels

  • S-box vs T-tables: T-tables 1.5–2× faster; index-dependent shared-memory access introduces a bank-pattern timing channel
  • Bitsliced (planned, time-boxed): constant-time by construction; 32 nodes/warp; not yet shipped — currently falls through to T-tables
  • Fixed-key: no measurable speedup over var-key (key schedule is already cheap relative to 10 rounds); would matter more for bitsliced
  • Memory mode: full vs rolling vs leaves-only is planned future work

Conclusion

  • GGM construction parallelizes naturally onto GPU per-level kernels
  • GPU T-tables: ~50M leaves/s @ $d{=}20$ — 6× over CPU AES-NI
  • AES >> Spongent on commodity GPU (~25×) — lightweight ≠ fast
  • Code, plots, full report at github.com/…/ggm-tree

Future work

  • Real warp-level bitsliced AES (currently falls through to T-tables)
  • Rolling-buffer and leaves-only memory modes
  • Spongent paper cross-validation
  • Real side-channel timing histograms
